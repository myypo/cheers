# Quieter

Reduce visual noise without making the interface generic. Quiet design depends on precision.

## Register

- **Product**: reduce decoration, motion, color, and competing affordances so the task is easier.
- **Brand**: lower intensity while preserving a specific point of view.

## Assess

Find intensity sources:

- too many saturated colors
- too many borders, shadows, backgrounds, cards, or icons
- all elements competing for attention
- dramatic motion where users are trying to work
- dense copy or repeated explanations
- inconsistent component treatments

## Refine

### Color

- Move product UI toward Restrained.
- Reserve accent for primary action, active state, focus, or semantic status.
- Desaturate brand colors rather than replacing them with gray.
- Use tinted neutrals, not pure gray, black, or white.
- Keep contrast strong enough for text and controls.

### Type and hierarchy

- Reduce excessive weights and sizes, but keep clear hierarchy.
- Let spacing and alignment carry more emphasis.
- Trim copy before shrinking it.

### Surfaces

- Remove unnecessary card wrappers.
- Flatten shadows unless elevation communicates interaction/state.
- Replace heavy borders with spacing, tonal contrast, or hairlines.

### Motion

- Remove decorative motion.
- Shorten distances and durations.
- Keep only feedback, reveal, loading, or transition motion.
- Respect reduced motion.

## Cheers fit

Quieting must not hide state clarity. Keep pending, error, disabled, focus-visible, and semantic status treatments visible. Do not hide backend-confirmed failures to make the UI calmer.

## Verify

The result should be easier to scan, still distinctive, still accessible, and still complete across states. If it became vague, run `clarify` or `layout` next.
