# Teach

Gather project design context and write strategic docs for future Cheers UI work.

## Outputs

- `PRODUCT.md`: strategy, users, purpose, register, personality, anti-references, design principles, accessibility.
- `DESIGN.md`: visual system, generated by `document` when enough visual information exists or seeded when the project is new.

## Step 1: inspect current context

Check for existing `PRODUCT.md`, `DESIGN.md`, README, docs, examples, app routes, components, CSS, and assets. Never silently overwrite existing docs.

Infer a register hypothesis:

- **brand**: landing, marketing, public docs/storytelling, portfolio, campaign.
- **product**: app shell, dashboard, settings, admin, forms, data tools.

The register can be overridden per task later, but PRODUCT.md should record the default.

## Step 2: ask only what is missing

Ask 2-3 questions per round and wait. Do not turn a sparse prompt into a finished PRODUCT.md without at least one user-answer round unless the repo docs fully answer everything.

Cover:

- register confirmation
- target users and context
- product purpose or brand promise
- desired emotional tone
- 2-3 references or anti-references
- accessibility and inclusion requirements
- strategic principles, not colors or fonts

Visual choices belong in DESIGN.md, not PRODUCT.md.

## Step 3: write PRODUCT.md

Use this structure:

```markdown
# Product

## Register

product

## Users

[Who they are, context, job to be done.]

## Product Purpose

[What this does, why it exists, what success looks like.]

## Brand Personality

[Voice, tone, emotional goals, 3 concrete personality words.]

## Anti-references

[Specific patterns, products, or aesthetics to avoid.]

## Design Principles

[3-5 strategic principles.]

## Accessibility & Inclusion

[WCAG target, reduced motion, color, keyboard, assistive tech, language needs.]
```

The register value should be exactly `product` or `brand`.

## Step 4: offer DESIGN.md

If UI code exists, offer to run `document` in scan mode. If the project is new, offer seed mode. Do not write DESIGN.md unless the user agrees or explicitly asked for it.

## Step 5: wrap up

Summarize what was written and the principles that future `cheers-design` commands should follow. If this teach flow was a blocker for `craft`, resume into `shape`; teach context is not a task-specific brief.
